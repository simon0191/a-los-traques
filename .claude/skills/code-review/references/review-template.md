# Review Template

The format used for PR reviews in this repo. Match it closely — the summary table and severity tiers are load-bearing for multi-round reviews.

## Structure

```markdown
## Review

[One-sentence acknowledgment of what the PR does / does well. Keep it short.]

### Critical

1. **[Short title]** (`path/to/file.js:LINE`)
   [1-3 sentences: what's wrong and why it matters. Include a code suggestion when the fix isn't obvious.]

   ```js
   // Suggested:
   ...
   ```

2. **[Next critical item]** (`path/to/file.js:LINE`)
   ...

### Moderate

1. **[Title]** (`path/to/file.js:LINE`)
   [Explanation.]

### Minor

- **[Title]** (`path/to/file.js:LINE`) — [Brief one-liner, these don't need much.]

### Nits

- [Even briefer; one line each.]

### Summary

| Issue | Severity | Action |
|-------|----------|--------|
| Missed winnerIndex in onMatchOver | Critical | Fix before merge |
| Brittle SQL substring assertion | Moderate | Refactor test |
| Unused import in FightScene | Minor | Remove |
| Comment phrasing in Logger.js | Nit | Optional |

[Closing line: "Changes requested" / "LGTM pending fixes" / "Ship it" / "Approved with nits".]

---
_Generated by [Claude Code](https://claude.ai/code)_
```

## Worked example

Here's what a realistic review for this repo looks like. This is synthetic but mirrors the shape of real reviews in recent PRs.

```markdown
## Review

Good catch on the duplicate win recording — the unified `onMatchOver` handler is the right direction. A few things to address before merge.

### Critical

1. **`winnerIndex` not passed in the main call site** (`src/scenes/FightScene.js:612`)
   The change to `onMatchOver(winnerIndex)` is correct in the `round_ko` branch, but the main path in `update()` still calls `this.onMatchOver()` without arguments, so the local KO flow records a loss for whoever won. Same bug this PR was meant to fix, still present in the path that fires 99% of the time.

   ```js
   // FightScene.js:612, current:
   this.onMatchOver();
   // Suggested:
   this.onMatchOver(winnerIndex);
   ```

2. **Rollback snapshot missing new `sim.lastHitFrame` field** (`src/simulation/FighterSim.js:48`, `src/systems/RollbackManager.js:captureSnapshot`)
   New field on `FighterSim` isn't in the snapshot helper. On rollback, it resets to 0 instead of restoring, which will show up as inconsistent combo timing across peers after any misprediction. Add it to both `captureSnapshot` and `restoreSnapshot`.

### Moderate

1. **Test asserts on SQL substring rather than behavior** (`tests/api/fights.test.js:92`)
   `expect(query).toContain('ORDER BY created_at DESC')` breaks the next time we touch the query, even for equivalent changes. Extract the query into an exported function (`buildRecentFightsQuery`) and test that the handler uses it + what it returns. Same pattern as `api/leaderboard.js`.

2. **Scope drift into `PartyKit` logging** (`party/server.js:184-211`)
   The structured logging cleanup is useful but unrelated to the win/loss fix. Would land cleaner as its own PR — easier to revert if the reconnection flow regresses and we need to bisect.

3. **Claimed test is missing from the diff** (PR description)
   The description says "also adds unit test for VictoryScene" but there's no `tests/scenes/VictoryScene.test.js` or similar in the diff. Either add the test or edit the description — this has been a recurring pattern worth cutting off.

4. **New pure logic without tests** (`src/services/TournamentManager.js:180-235`)
   The new `seedHumansApart` function is pure (deterministic, no Phaser) and non-trivial — seeding a bracket so N humans meet as late as possible has real branching behavior. Pure logic is the whole reason `TournamentManager` lives in `services/` rather than a scene; it needs a test in `tests/systems/` or `tests/services/` covering at least 2-human, 4-human, and odd-player-count cases. Without one, the next refactor will silently break the bracket shape and nobody will notice until a tournament goes sideways.

### Minor

- **Leftover `console.log`** (`src/scenes/VictoryScene.js:58`) — Replace with `log.debug` or remove.
- **Unreachable branch** (`src/services/TournamentManager.js:241`) — `if (this.humanFighterIds.length === 0)` can't be true given the guard at 237; dead code.

### Summary

| Issue | Severity | Action |
|-------|----------|--------|
| winnerIndex missing in main onMatchOver | Critical | Fix before merge |
| Snapshot missing lastHitFrame | Critical | Add to capture/restore |
| Claimed test missing from diff | Critical | Add the test or edit the description |
| New pure logic in TournamentManager without tests | Critical | Add unit tests |
| SQL substring assertion | Moderate | Refactor test |
| Scope drift into PartyKit logging | Moderate | Split into separate PR (your call) |
| console.log in VictoryScene | Minor | Remove |
| Unreachable branch in TournamentManager | Minor | Remove |

Changes requested. Two criticals block merge; the moderates are your call but worth addressing.

---
_Generated by [Claude Code](https://claude.ai/code)_
```

## Multi-round reviews

For follow-up reviews (v2, v3, round 2...), keep a running status table at the top showing what's been addressed:

```markdown
## Review — v2

Status of items from the previous review:

| Issue | Previous | v2 |
|-------|----------|-----|
| winnerIndex in main onMatchOver | Critical | ✅ Fixed |
| Snapshot missing lastHitFrame | Critical | ✅ Fixed |
| SQL substring assertion | Moderate | ❌ Not addressed |
| Scope drift into PartyKit logging | Moderate | ✅ Split to #127 |
| console.log in VictoryScene | Minor | ✅ Removed |

### Still outstanding

- [Re-raise unaddressed items from last round, with any new context.]

### New in v2

- [Any new issues introduced by the latest push.]
```

This makes it easy to see at a glance whether the PR is converging or not, and prevents items from silently dropping off the list.

## Approval-only reviews

For PRs that are clean on a re-review:

```markdown
## Review — v3

All items from v2 addressed. Clean follow-up — ship it.

---
_Generated by [Claude Code](https://claude.ai/code)_
```

No need to pad a clean review with false issues.

## Tiny PRs

For small bug fixes that don't need the full structure:

```markdown
## Review

Fix looks right — matches the flow in `onRoundEnd` and I checked the other call sites, no regressions. LGTM.

---
_Generated by [Claude Code](https://claude.ai/code)_
```

Matching the review depth to the change size is part of writing a good review.
